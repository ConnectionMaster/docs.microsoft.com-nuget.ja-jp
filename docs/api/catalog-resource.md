<span data-ttu-id="fac2e-422">クライアントを格納する必要があります、カタログは、時間インデックスを追加専用のデータ構造体であるため、**カーソル**まで、どの時点でクライアントを表すが処理されるカタログ アイテムがローカルにします。</span><span class="sxs-lookup"><span data-stu-id="fac2e-422">Because the catalog is an append-only data structure indexed by time, the client should store a **cursor** locally, representing up to what point in time the client has processed catalog items.</span></span> <span data-ttu-id="fac2e-423">このカーソル値を生成しないこと、クライアントのマシンのクロックを使用してに注意してください。</span><span class="sxs-lookup"><span data-stu-id="fac2e-423">Note that this cursor value should never be generated using the client's machine clock.</span></span> <span data-ttu-id="fac2e-424">代わりに、値の元のカタログ オブジェクトの`commitTimestamp`値。</span><span class="sxs-lookup"><span data-stu-id="fac2e-424">Instead, the value should come from a catalog object's `commitTimestamp` value.</span></span>

<span data-ttu-id="fac2e-425">たびに、クライアントがパッケージ ソースの新しいイベントを処理しようとすると、その必要がありますカタログのクエリのみ、コミット タイムスタンプのすべてのカタログ アイテムのストアド カーソルより大きい。</span><span class="sxs-lookup"><span data-stu-id="fac2e-425">Every time the client wants to process new events on the package source, it need only query the catalog for all catalog items with a commit timestamp greater than its stored cursor.</span></span> <span data-ttu-id="fac2e-426">クライアントでは、すべての新しいカタログ項目が正常に処理、後に新しいカーソル値としてだけ処理されたカタログ項目の最新のコミット タイムスタンプを記録します。</span><span class="sxs-lookup"><span data-stu-id="fac2e-426">After the client successfully processes all new catalog items, it records the latest commit timestamp of catalog items just processed as its new cursor value.</span></span>

<span data-ttu-id="fac2e-427">このアプローチを使用して、クライアントはパッケージ ソースで発生したすべてのパッケージ イベントを見逃すことはありませんを確認できます。</span><span class="sxs-lookup"><span data-stu-id="fac2e-427">Using this approach, the client can be sure to never miss any package events that occurred on the package source.</span></span>
<span data-ttu-id="fac2e-428">さらに、クライアント カーソルの記録されたコミット タイムスタンプより前の古いイベントを再処理する必要はありません。</span><span class="sxs-lookup"><span data-stu-id="fac2e-428">Additionally, the client never has to reprocess old events prior to the cursor's recorded commit timestamp.</span></span>

<span data-ttu-id="fac2e-429">このカーソルの強力な概念は、nuget.org のバック グラウンド ジョブの多くは使用され、V3 API 自体の最新の状態を維持するために使用します。</span><span class="sxs-lookup"><span data-stu-id="fac2e-429">This powerful concept of cursors is used for many of nuget.org background jobs and is used to keep the V3 API itself up-to-date.</span></span>
